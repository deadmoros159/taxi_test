# Архитектура Taxi Service

## Микросервисы

### 1. Auth Service (порт 8000)
**Ответственность:** Аутентификация, авторизация, управление пользователями

**Основные функции:**
- Регистрация и авторизация пользователей (SMS/Email коды)
- Управление JWT токенами (access/refresh)
- Управление ролями пользователей
- Регистрация диспетчеров и админов

**Роли:**
- `passenger` - обычный пользователь
- `driver` - водитель
- `dispatcher` - диспетчер
- `admin` - администратор

### 2. Driver Service (порт 8001)
**Ответственность:** Управление водителями и автомобилями

**Основные функции:**
- Регистрация водителей (диспетчерами/админами)
- Хранение данных водителей (права, паспорт, фото)
- Хранение данных автомобилей (марка, модель, гос. номер, VIN, фото)
- Управление статусами водителей
- Верификация водителей

### 3. Order Service (порт 8002)
**Ответственность:** Управление заказами, долгами, WebSocket уведомления

**Основные функции:**
- Создание заказов (пассажирами)
- Принятие заказов (водителями)
- Отмена заказов (с обязательной причиной)
- Расчет времени до клиента
- Система долгов водителей (20% с каждого заказа)
- WebSocket для real-time уведомлений

**Статусы заказов:**
- `pending` - ожидает водителя
- `accepted` - принят водителем
- `driver_arrived` - водитель прибыл
- `in_progress` - в пути
- `completed` - завершен
- `cancelled` - отменен
- `cancelled_by_driver` - отменен водителем
- `cancelled_by_passenger` - отменен пассажиром

### 4. Admin Service (порт 8003)
**Ответственность:** Агрегация данных, статистика, дашборды

**Основные функции:**
- Агрегация данных из других сервисов
- Статистика по заказам, водителям, пользователям
- Дашборды для админов и диспетчеров
- Отчеты за периоды

**Принцип работы:**
- Admin Service НЕ дублирует бизнес-логику
- Он агрегирует данные из других сервисов
- Операции (создание, изменение) делегируются в доменные сервисы

### 5. API Gateway (порт 8080)
**Ответственность:** Единая точка входа, маршрутизация запросов

## Геолокация

### Текущее решение: Shared библиотека

Геолокация реализована как утилита в `shared/geolocation.py`:

**Функции:**
- `calculate_distance()` - расчет расстояния по формуле Haversine
- `estimate_travel_time()` - оценка времени в пути
- `calculate_route_time()` - расчет маршрута (расстояние + время)
- `is_point_in_radius()` - проверка точки в радиусе
- `find_nearest_point()` - поиск ближайшей точки

**Почему не отдельный микросервис?**

1. **Простота:** Формула Haversine - простая математика, не требует отдельного сервиса
2. **Производительность:** Локальные вычисления быстрее сетевых вызовов
3. **Нет состояния:** Геолокация - stateless функция
4. **Нет сложной логики:** Не нужны кэширование, очереди, БД

### Когда имеет смысл отдельный микросервис?

Отдельный `location-service` имеет смысл, если нужны:

1. **Сложные расчеты:**
   - Построение маршрутов через Google Maps API
   - Оптимизация маршрутов (TSP, VRP)
   - Расчет стоимости по тарифам

2. **Кэширование:**
   - Кэш маршрутов
   - Кэш геокодирования адресов
   - Кэш обратного геокодирования

3. **Интеграции:**
   - Google Maps API
   - Yandex Maps API
   - OpenRouteService
   - Mapbox

4. **Масштабирование:**
   - Огромное количество запросов
   - Сложные алгоритмы (машинное обучение для прогноза времени)

**Рекомендация:** Начните с shared библиотеки, выносите в отдельный сервис при необходимости.

## WebSocket

### Текущее решение: В Order Service

WebSocket реализован в `order-service` через `WebSocketManager`:

**Функции:**
- Уведомления водителей о новых заказах
- Обновления заказов для пассажиров
- Heartbeat для поддержания соединений

**Почему в Order Service?**

1. **Тесная связь:** WebSocket уведомления связаны с заказами
2. **Простота:** Не нужна дополнительная инфраструктура
3. **Меньше задержек:** Прямой доступ к данным заказов

### Когда имеет смысл отдельный микросервис?

Отдельный `websocket-service` имеет смысл, если:

1. **Кластер:** Нужен Redis Pub/Sub для синхронизации между инстансами
2. **Масштаб:** Огромное количество одновременных соединений
3. **Специализация:** Команда фокусируется только на real-time
4. **Изоляция:** Проблемы с WebSocket не должны влиять на основной API

**Рекомендация:** Оставьте в Order Service, выносите при необходимости кластера.

## Базы данных

- `auth_db` - пользователи, токены (auth-service)
- `driver_db` - водители, автомобили (driver-service)
- `order_db` - заказы, долги (order-service)
- `redis` - кэш, rate limiting (auth-service)

## Связи между сервисами

```
API Gateway
    ↓
    ├──→ Auth Service (проверка токенов)
    ├──→ Driver Service (данные водителей)
    ├──→ Order Service (заказы, WebSocket)
    └──→ Admin Service (статистика, дашборды)

Admin Service
    ├──→ Auth Service (получение пользователей)
    ├──→ Driver Service (получение водителей)
    └──→ Order Service (получение заказов)

Order Service
    ├──→ Auth Service (проверка токенов)
    └──→ Driver Service (данные водителей для заказов)
```

## Принципы архитектуры

1. **Domain-Driven Design:** Каждый сервис отвечает за свой домен
2. **API Gateway:** Единая точка входа
3. **Stateless:** Сервисы не хранят состояние между запросами
4. **Loose Coupling:** Минимальная связанность между сервисами
5. **Single Responsibility:** Каждый сервис делает одну вещь хорошо

## Масштабирование

- **Auth Service:** Масштабируется по количеству пользователей
- **Driver Service:** Масштабируется по количеству водителей
- **Order Service:** Масштабируется по количеству заказов (может потребоваться Redis для WebSocket)
- **Admin Service:** Масштабируется по нагрузке дашбордов

## Безопасность

- JWT токены для аутентификации
- Проверка ролей в каждом сервисе
- Rate limiting в auth-service
- CORS настройки для каждого сервиса
